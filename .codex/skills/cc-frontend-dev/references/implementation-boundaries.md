# 前端实现边界

## `try/catch`

- 以下场景通常不需要 `try/catch`：条件渲染、普通事件处理、状态更新、简单映射和格式化、类型已明确的 props/state 读取。
- 以下场景通常需要边界处理：请求、`JSON.parse`、Storage 读写、URL 参数解析、动态导入、第三方 SDK、可能抛错的浏览器 API。
- `catch` 不能吞错；没有明确恢复策略时，优先保留错误可见性和上下文，而不是假装成功。

## 容灾状态

- 默认沿用页面已有的 `loading / empty / error` 模式。
- 没有明确需求或现有模式时，不主动补齐完整容灾链路。
- 为修单点问题时，只补与当前路径直接相关的最小状态，不顺手扩成全页兜底方案。

## 抽象和封装

- 单文件、单次使用且紧贴组件语义的小逻辑，优先留在组件内。
- 只有满足以下任一条件时才抽离：已经有多个真实调用点、需要隔离副作用、需要跨组件共享稳定语义、单独测试能明显降低主流程理解成本。
- 不为减少几行重复、命名整洁感或假想复用抽 `helper`、`wrapper`、`utils`。
