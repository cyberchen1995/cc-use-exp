## 处理优先级

### P0

- 明确仓库真正的权利主体是谁
- 统一维护者身份、GitHub 账号、对外授权邮箱和未来签署主体
- 确认 `wwj` 与 `doccker` 是否属于同一控制主体

### P1

- 逐个联系非维护者贡献者，补签 [CLA.md](../../CLA.md) 或其他单独授权文件
- 对无法联系、拒绝签署或权属不清的贡献，识别其涉及的文件范围和改动范围
- 对关键文件中的历史外部贡献，优先评估是否需要重写

### P2

- 将已签署并归档的贡献者加入 `.github/cla/allowlist.txt`
- 对一次性或低频贡献者，保留仓外签署记录，不必默认加入白名单
- 定期复查是否还有新贡献在未签署状态下被合并

## 推荐执行步骤

1. 生成按作者聚合的文件范围清单
2. 标记哪些贡献是琐碎修正，哪些是实质性创作
3. 对实质性创作优先补签或重写
4. 将联系记录、签署状态、替代重写状态沉淀到内部表格
5. 只有在权利链清晰后，再推进更强的商业化动作

## 建议维护的内部台账字段

- 贡献者姓名 / 实体名称
- GitHub 账号
- 邮箱
- 提交 hash 范围
- 涉及文件
- 是否属于实质性创作
- 是否已联系
- 是否已签署 `CLA`
- 签署日期
- 归档位置
- 是否已加入 allowlist
- 若未签署，是否已重写替换

## 处理原则

- 不要只按提交数判断风险，少量但核心的提交也可能构成高风险
- 文档、图片、模板和脚本同样可能涉及版权和商标风险
- 对来源不明或无法联系的贡献，重写通常比长期带着不确定权利链更稳
- 如果未来考虑成立公司主体或切换对外授权主体，应尽早统一签署方
