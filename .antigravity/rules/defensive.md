# 防御性编码与质量规则

> 预防常见 AI 编码陷阱，基于工程最佳实践。

<skill_rules>
## 🤝 AI 协作三原则
> 元规则：规范 AI 如何与用户协作，先于具体禁止/必须项。

### 1. 用户主权（Locus of Control）
- ❌ 擅自扩需求、主动重构未要求的代码
- ❌ 跳过用户确认执行不可逆操作
- ✅ 复杂任务先说计划再实现；危险操作必须确认

### 2. 状态显式化（Reduce Memory Load）
- ❌ 让用户自己记住"刚才说过什么"
- ❌ 任务中途无任何状态可视化
- ✅ 多步任务用 TODO 列表；功能开发显式记录任务文件

### 3. 输出一致性（Consistency）
- ❌ 同类操作输出格式各不相同
- ❌ 同义命令命名混乱
- ✅ 规则溯源统一格式；命令命名遵循现有约定
</skill_rules>

<anti_patterns>
## 🚫 严格禁止
### 1. 测试篡改（最严重）
- ❌ 修改测试断言来适配错误实现
- ❌ 删除或注释掉失败的测试
- ✅ 测试失败时，必须修复实现代码，保持测试集的完整性

### 2. 过度工程化
- 简单 if/else 优于复杂抽象
- 不要引入未明确要求的模式（如“工厂”、“策略”），除非项目已存在大规模同类用法

### 3. 盲目修改代码
修改前必须：
- 搜索类似功能，了解项目现有实现模式
- 检查修改点的所有上游调用，评估回归风险

### 3.5 重构代码（高风险）
**重构前必须**：
- 完整读取原始代码（不凭记忆或推测）
- 制作对比清单（表格列/数据字段/配置项）
- 逐项验证一致性（数量/顺序/命名）

**表格重构**：列数、列名、列顺序必须与原始代码完全一致
**条件分支重构**：所有 if/else/switch 分支都必须处理，不能只处理一个分支就推测其他分支
**服务拆分重构**：先提取共享工具方法到独立类，再拆分子服务（防止循环依赖）

### 4. 提交物质量
- ❌ 提交调试残留（`console.log`, `print`, `fmt.Println`）
- ❌ 用“注释掉旧实现”代替删除代码
- ✅ 所有的 `TODO` 必须在任务结束前清理或转为正式 issue

### 5. 自动添加 AI 元数据
- ❌ 禁止添加 "Generated by Gemini" 或 AI 自动签名
- ❌ 禁止添加冗余的变更总结文档（除非明确要求）

### 6. 用"全局开关"掩盖错误（最隐蔽）

**症状**：报错出现 → 反应是去 `application.properties` / `tsconfig` / `build.gradle`
开一个全局放行开关让报错消失，而不是修根因。

**典型反模式**：

| 语言/框架 | 错误开关 | 真正应该做的 |
|----------|---------|------------|
| Spring Boot | `spring.main.allow-bean-definition-overriding=true` | 找重名 Bean，改 name 或拆 Bean |
| Spring Boot | `spring.main.allow-circular-references=true` | 找循环依赖根因，拆 Service / `@Lazy` |
| TypeScript | `skipLibCheck` / `noImplicitAny:false` / `strict:false` | 修类型错误 / 补 `.d.ts` |
| TypeScript | `// @ts-ignore` / `// @ts-nocheck` 整文件 | 针对单行修正类型 |
| Maven | `-Dmaven.test.skip` / `-DskipTests` | 修失败的测试 |
| Gradle | 禁用 Test 任务 | 修失败的测试 |
| Go build | `// +build ignore` 全文件忽略 | 拆出有问题的代码 |
| Python | `warnings.filterwarnings('ignore')` 全局 | 针对单个 warning 处理 |
| Webpack/Vite | `transpileOnly: true` | 修类型，不要绕过 |
| ESLint | 文件顶 `/* eslint-disable */` | 针对单条规则 disable |
| Hibernate | `hbm2ddl.auto=update` 掩盖 schema 漂移 | 用迁移工具（Flyway/Liquibase） |

**判定原则**：
- 启用任何 `allow-*` / `ignore-*` / `skip-*` / `disable-*` 配置前，你 **必须** 先使用 `<diagnostic>` 标签进行强制自省：
  <diagnostic>
  1. 真实的底层报错信息是什么？
  2. 产生报错的根本代码原因是什么（绝不是因为没开开关）？
  3. 如果开启该开关，未来会被掩盖的潜在业务风险是什么？
  </diagnostic>
- 在 `<diagnostic>` 分析完成前，禁止修改全局配置文件。
- 临时绕过单行可接受（必须带 TODO 和 issue 链接）；改全局配置直接驳回
- 每个掩盖式开关都意味着"以后还会有相同根因的 bug"

**与第 1 条「测试篡改」的关系**：本条是"配置层"的测试篡改 ——
不改测试断言，但改运行时让错误不抛出来，本质一样。
</anti_patterns>

<workflow>
## ✅ 架构与流程
### 1. Type-First (类型先行)
无论是编写新业务还是**修改现有逻辑**，必须遵循 **“修改数据结构 -> 修改逻辑 -> 修改调用/模板”** 的单向防呆流程：
- **定义先行**：编写前必须先定义（或更新）数据结构：
  - **Go**: struct / interface
  - **TypeScript**: interface / type
  - **Java**: DTO / VO / Entity
- **严禁越级调用**：禁止在未更新类型定义的情况下，凭直觉在前端模板或业务逻辑中调用不存在的字段（如 `user.nickname`），防范“幻觉属性”导致的编译失败。

### 2. 复杂任务分步确认
当任务涉及超过 3 个文件或修改核心逻辑时，必须：
- 步骤 1：调用 `update_topic` 声明方案计划，说明影响。
- 步骤 2：使用 `ask_user` 工具（或在聊天中明确询问）等待用户确认 ⏸️
- 步骤 3：获得授权后，再分步实施。
</workflow>

<decision_principles>
## 📂 决策原则
```
简单方案 > 复杂方案
复用现有 > 创建新的
直接实现 > 抽象封装
排查: 复现 -> 假设 -> 验证 -> 最小修复
```

**HTTP 错误诊断**：当第三方 API 返回 HTTP 错误且 body 为空、headers 极简（缺少目标 API 的标准 headers），优先怀疑 CDN/WAF/代理层拦截，而非 API 本身的业务错误。验证方法：同一 API 的 GET 请求是否正常（GET 正常 POST 异常 → HTTP 客户端兼容性问题）。
</decision_principles>
